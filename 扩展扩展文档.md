  你期望的架构

  应该是：
  用户输入 → 意图识别 → Agent 获得：
                          ├─ 身份证号
                          ├─ 用户需求
                          ├─ 判定条件清单（来自政策JSON）
                          └─ Text-to-SQL Tool（MCP）
                               ↓
                          Agent 自主决策：
                          ├─ 我需要查什么？（生成 Text）
                          ├─ 调用 Text-to-SQL Tool
                          ├─ 获取结果
                          └─ 基于结果继续推理/辩论

  关键转变：
  - Text-to-SQL 从"前置批处理模块" → "Agent 可调用的 MCP Tool"
  - 证据从"静态打包" → "动态按需查询"

  三层条件分类

  你提出的分类很关键：

  ┌──────────────┬─────────────────────────────────────┬────────────────────────────────────┐  
  │   条件类型   │              处理方式               │                示例                │  
  ├──────────────┼─────────────────────────────────────┼────────────────────────────────────┤  
  │ 硬性必须满足 │ 程序直接调用                        │ 生存状态='生存'                    │  
  │              │ Text-to-SQL，不通过则直接拒绝       │                                    │  
  ├──────────────┼─────────────────────────────────────┼────────────────────────────────────┤  
  │ 硬性必须排除 │ 程序直接调用                        │ 不能是个体工商户                   │  
  │              │ Text-to-SQL，命中则直接拒绝         │                                    │  
  ├──────────────┼─────────────────────────────────────┼────────────────────────────────────┤  
  │ 需要灵活评判 │ Agent 自主调用                      │ 距离退休年龄、收入波动、缴费连续性 │  
  │              │ Text-to-SQL，获取原始数据后推理     │                                    │  
  └──────────────┴─────────────────────────────────────┴────────────────────────────────────┘  

  这意味着：
  - 政策 JSON 需要标注每个条件的"类型"（rule_type: "hard_pass" / "hard_exclude" / "flexible"） 
  - 硬性条件在 Agent 辩论前就执行（快速失败）
  - 灵活条件由 Agent 按需查询原始数据后自主判断

  ---
  衍生问题：5个Agent会重复查询吗？

  你担心的"5个Agent各自调用Text-to-SQL查询5次"确实低效。

  可能的优化方向（不是现在的解决方案，只是方向）：
  1. 共享查询缓存：同一 session 内，相同 SQL 只执行一次
  2. 证据池机制：Agent 查询的结果写入共享证据池，其他 Agent 可见
  3. 分阶段执行：
    - 第一阶段：程序执行硬性条件（快速失败）
    - 第二阶段：Agent 按需查询灵活条件（可能重复，但有缓存）
    - 第三阶段：Agent 基于已有证据辩论
