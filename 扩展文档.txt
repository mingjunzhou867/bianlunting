# 多政策意图理解系统 - 详细技术方案

> 文档版本: v1.0
> 创建日期: 2026-03-25
> 目标: 从单政策系统升级为"身份证号+自然语言需求"的多政策智能核查系统

---

## 一、需求概述

### 1.1 核心变化

**输入变化**:
- 旧: `id_card` (身份证号)
- 新: `id_card` + `user_query` (自然语言需求)

**示例输入**:
```
身份证号: 420102199001011234
需求: "判断这个人是否有资格领取失业补贴"
```

```
身份证号: 420102199001011234
需求: "这个人能不能申请灵活就业社保补贴?"
```

```
身份证号: 420102199001011234
需求: "看看这个人适不适合做主动服务"
```

### 1.2 涉及的三大政策

1. **灵活就业社保补贴** (当前已实现)
   - 资格认定 + 金额计算 + 月数计算
   
2. **失业补贴** (新增)
   - 失业状态认定 + 缴费年限核查 + 领取历史查询
   
3. **企业社保补贴主动服务** (新增)
   - 增员扫描 + 疑似困难人员识别 + 前置条件检查

## 二、三个关键问题的技术决策

### 问题1: 口语化输入的二次确认,是否需要智能体?

#### ✅ 推荐方案: **需要一个专门的"意图理解Agent"**

**理由**:
1. **炫技价值高**: 展示多Agent协作能力,不只是辩论Agent,还有前置理解Agent
2. **可解释性强**: Agent可以输出结构化的理解结果,包含置信度、歧义点、需要确认的内容
3. **扩展性好**: 未来可以处理更复杂的复合需求 (如"这个人能领哪些补贴?")

**Agent职责**:
```python
class IntentUnderstandingAgent:
    """
    意图理解Agent - 负责将口语化需求转为结构化政策意图
    """
    
    输入: 
        user_query: "这个人能不能申请灵活就业社保补贴?"
    
    输出:
        PolicyIntent {
            policy_id: "POLICY_001",
            policy_name: "灵活就业社保补贴",
            action_type: "资格认定",
            confidence: 0.95,
            ambiguities: [],
            need_confirmation: false
        }
    
    输入 (歧义案例):
        user_query: "看看这个人的补贴情况"
    
    输出:
        PolicyIntent {
            policy_id: null,
            policy_name: null,
            action_type: "资格认定",
            confidence: 0.3,
            ambiguities: [
                "未明确指定补贴类型",
                "可能指: 灵活就业补贴、失业补贴、企业补贴"
            ],
            need_confirmation: true,
            candidate_policies: [
                {"policy_id": "POLICY_001", "policy_name": "灵活就业社保补贴"},
                {"policy_id": "POLICY_002", "policy_name": "失业补贴"}
            ]
        }
```

**前端交互流程**:
```
用户输入 → IntentUnderstandingAgent解析 
         ↓
    [置信度 >= 0.8]
         ↓
    直接进入取证辩论流程
    
用户输入 → IntentUnderstandingAgent解析 
         ↓
    [置信度 < 0.8]
         ↓
    前端弹出确认框:
    "系统理解您想查询: 灵活就业社保补贴
     是否正确? 
     [✓ 是的] [✗ 不是,我想查失业补贴] [✗ 我想查所有补贴]"
         ↓
    用户确认后进入取证辩论流程
```

**Agent实现要点**:
- **Temperature**: 0.2 (需要稳定输出,不需要创造性)
- **System Prompt**: 包含政策列表、关键词映射、常见口语化表达
- **输出格式**: 强制JSON Schema,确保结构化
- **Few-shot示例**: 提供10-15个典型案例

### 问题2: 政策解析方案选择

#### ✅ 推荐方案: **混合方案C (数据库 + MCP智能体)**

**方案对比**:

| 维度 | 方案A: 纯数据库 | 方案B: 纯MCP智能体 | 方案C: 混合方案 (推荐) |
|------|----------------|-------------------|---------------------|
| 炫技感 | ⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 可维护性 | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 成本 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| 适合毕设 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |

**混合方案C的架构**:

```
政策存储层 (数据库)
├── policy_registry (政策注册表)
│   ├── policy_id, policy_name, policy_type
│   ├── policy_text (政策原文,存储为TEXT)
│   └── structured_rules (结构化规则,存储为JSON)
│
└── policy_qualification_items (资格项表)
    ├── item_id, policy_id, item_name
    └── check_logic (检查逻辑描述)

政策理解层 (MCP智能体)
├── PolicyParserAgent (政策解析Agent)
│   ├── 输入: policy_text (政策原文)
│   ├── 输出: structured_rules (结构化规则)
│   └── 触发时机: 新政策录入时 (一次性)
│
└── PolicyReasoningAgent (政策推理Agent)
    ├── 输入: policy_id + evidence_bundle
    ├── 输出: policy_specific_judgment
    └── 触发时机: 每次辩论时 (实时)
```

**为什么推荐混合方案?**

1. **兼顾炫技与实用**:
   - 政策录入时调用`PolicyParserAgent`,展示"AI理解政策文本"的能力
   - 运行时直接读数据库,保证性能
   
2. **展示两种Agent应用场景**:
   - **离线Agent**: PolicyParserAgent (政策解析,一次性任务)
   - **在线Agent**: IntentUnderstandingAgent + 5个辩论Agent (实时任务)
   
3. **论文价值高**:
   - 可以写"基于Agent的政策知识抽取"章节
   - 可以写"多Agent协作的政务决策系统"章节
   
4. **演示效果好**:
   - 可以在前端展示"政策解析过程"
   - 可以对比"人工录入规则 vs AI解析规则"的准确率

### 问题3: 政策存储方式

#### ✅ 推荐方案: **数据库为主 + JSON文件为辅**

**存储策略**:

```
政策核心数据 → MySQL数据库
├── policy_registry (政策主表)
├── policy_qualification_items (资格项表)
└── policy_rules (规则明细表)

政策辅助数据 → JSON文件 (类似dicts/)
├── policies/POLICY_001_灵活就业补贴.json
├── policies/POLICY_002_失业补贴.json
└── policies/POLICY_003_主动服务.json
```

**为什么不全放JSON?**

| 数据类型 | 存储位置 | 理由 |
|---------|---------|------|
| 政策元数据 (id, name, type, 生效日期) | 数据库 | 需要查询、筛选、关联 |
| 政策原文 (policy_text) | 数据库TEXT字段 | 需要版本管理、审计 |
| 结构化规则 (structured_rules) | 数据库JSON字段 | 需要事务一致性 |
| 政策完整快照 (含注释、示例) | JSON文件 | 便于版本控制、人工审阅 |
| 政策关键词映射 | JSON文件 | 供IntentAgent加载 |

**为什么不全放数据库?**

1. **JSON文件便于Git版本控制**: 政策变更可追溯
2. **JSON文件便于人工审阅**: 导师/评委可以直接打开查看
3. **JSON文件便于Agent加载**: 作为Prompt上下文注入更方便

**推荐的JSON文件格式** (参考BUSINESS_ROLE.json):


**数据库Schema**:

```sql
-- 政策注册表
CREATE TABLE policy_registry (
    policy_id VARCHAR(50) PRIMARY KEY,
    policy_name VARCHAR(200) NOT NULL,
    policy_type VARCHAR(100),
    effective_date DATE,
    expiry_date DATE,
    policy_text TEXT,  -- 政策原文
    structured_rules JSON,  -- 结构化规则 (与JSON文件同步)
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_policy_type (policy_type),
    INDEX idx_effective_date (effective_date)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='政策注册表';

-- 政策资格项表
CREATE TABLE policy_qualification_items (
    item_id VARCHAR(50) PRIMARY KEY,
    policy_id VARCHAR(50) NOT NULL,
    item_name VARCHAR(200) NOT NULL,
    item_type ENUM('基础条件', '排斥条件', '合理推断', '精算规则') NOT NULL,
    check_logic TEXT,  -- 检查逻辑描述
    sql_template_ref VARCHAR(50),  -- 关联的SQL模板ID
    FOREIGN KEY (policy_id) REFERENCES policy_registry(policy_id),
    INDEX idx_policy_id (policy_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='政策资格项表';

-- 政策关键词映射表 (用于意图识别)
CREATE TABLE policy_keywords (
    keyword_id INT AUTO_INCREMENT PRIMARY KEY,
    policy_id VARCHAR(50) NOT NULL,
    keyword VARCHAR(100) NOT NULL,
    keyword_type ENUM('正式名称', '别名', '口语化表达', '行业黑话') NOT NULL,
    weight DECIMAL(3,2) DEFAULT 1.0,  -- 权重,用于歧义消解
    FOREIGN KEY (policy_id) REFERENCES policy_registry(policy_id),
    INDEX idx_keyword (keyword),
    INDEX idx_policy_id (policy_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='政策关键词映射表';
```

### 3.2 核心数据流

```
用户输入: id_card="420102199001011234", user_query="这个人能领灵活就业补贴吗?"
    ↓
[步骤1: 意图理解]
IntentUnderstandingAgent.understand(user_query)
    → PolicyIntent {
        policy_id: "POLICY_001",
        policy_name: "灵活就业社保补贴",
        action_type: "资格认定",
        confidence: 0.95,
        need_confirmation: false
    }
    ↓
[步骤2: 政策路由]
PolicyRouter.route(policy_id="POLICY_001")
    → 从 policy_registry 读取 structured_rules
    → 从 policies/POLICY_001.json 读取完整配置
    → 生成 SemanticPacket (任务级语义包)
    ↓
[步骤3: 取证规划]
EvidencePlanner.plan(person_id, policy_id, semantic_packet)
    → EvidencePlan {
        questions: [
            {question_id: "Q001", description: "检查生存状态", ...},
            {question_id: "Q002", description: "检查户籍", ...}
        ],
        plan_items: [
            {plan_id: "P001", question_ref: "Q001", query_intent: {...}},
            ...
        ]
    }
    ↓
[步骤4: 动态取证]
DynamicEvidenceCollector.collect(evidence_plan)
    → 逐条执行 plan_items
    → Text2SQL生成SQL → 执行 → AutoDebugger修复 → EvidenceAssembler装配
    → EvidenceBundle (Evidence v2格式)
    ↓
[步骤5: 证据投影]
project_evidence(evidence_bundle)
    → EvidenceProjection (Agent友好格式)
    ↓
[步骤6: 多Agent辩论]
DebateOrchestrator.debate(evidence_projection, policy_context)
    → 5个通用Agent + 1个PolicyReasoningAgent (可选)
    → 独立判断 → 辩论 → 投票 → 共识
    → DebateResult
    ↓
[步骤7: 持久化]
persist_completed_session(debate_result)
    → 写入 debate_session 表
    → 写入 agent_debate_log 表
    ↓
[步骤8: 返回前端]
返回完整辩论结果 + 政策上下文
```

## 八、总结与建议

### 8.1 核心价值

这次升级的核心价值不只是"多了2个政策",而是:

1. **架构升级**: 从单政策硬编码 → 多政策可扩展架构
2. **交互升级**: 从结构化输入 → 自然语言交互
3. **智能升级**: 从单一辩论 → 意图理解+辩论+政策推理的多层智能
4. **展示升级**: 从"会辩论" → "会理解+会辩论+会学习"

### 8.2 最小可行方案 (MVP)

如果时间紧张,建议优先完成:

### 附录A: 失业补贴政策规则 (初稿)

```
失业补贴资格认定规则:

1. 基础条件 (必须全部满足):
   - 有失业登记记录
   - 失业前已缴纳失业保险满1年
   - 非本人意愿中断就业 (企业解聘/合同到期等)
   - 当前处于失业状态

2. 排斥条件 (触发任一条则不通过):
   - 当前有就业登记 (与灵活就业补贴互斥)
   - 已达到法定退休年龄
   - 正在领取养老保险待遇
   - 有骗保记录

3. 计算规则:
   - 可领取月数 = min(缴费年限对应月数, 24个月)
     - 缴费1-5年: 最多12个月
     - 缴费5-10年: 最多18个月
     - 缴费10年以上: 最多24个月
   - 月领取金额 = 当地最低工资标准 × 80%
```

```
企业社保补贴主动服务规则:
```

```python
# intent/models.py

from pydantic import BaseModel, Field
from typing import List, Optional

class PolicyIntent(BaseModel):
    """政策意图识别结果"""
    policy_id: Optional[str] = Field(None, description="政策ID")
    policy_name: Optional[str] = Field(None, description="政策名称")
    action_type: str = Field(..., description="行动类型: 资格认定/金额计算/历史查询/疑似人员识别")
    confidence: float = Field(..., ge=0.0, le=1.0, description="置信度")
    reasoning: str = Field(..., description="推理过程")
    ambiguities: List[str] = Field(default_factory=list, description="歧义点")
    need_confirmation: bool = Field(..., description="是否需要二次确认")
    candidate_policies: List[dict] = Field(default_factory=list, description="候选政策列表")

class PolicyConfig(BaseModel):
    """政策配置"""
    policy_id: str
    policy_name: str
    policy_type: str
    effective_date: str
    expiry_date: Optional[str]
    policy_text: str
    structured_rules: dict
    keywords: List[str]
    aliases: List[str]
    intent_patterns: List[str]
    evidence_plan_template: dict
    notes: List[str]
```
