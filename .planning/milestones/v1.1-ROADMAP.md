# Milestone v1.1: Dynamic Evidence Planning Design

**Status:** SHIPPED 2026-03-24  
**Phases:** 4-7  
**Total Plans:** 10

## Overview

This milestone did not ship new runtime behavior. It shipped the design contracts needed to evolve the project from a saved-session debate product into a database-aware, evidence-planning, dynamic-query system without losing the grounded `v1.0` baseline.

The main milestone outcome is a coherent implementation guide for the next generation of the architecture:

- one agreed five-layer system model
- formal database-cognition and dictionary-memory contracts
- formal evidence-planning, dynamic-query, and Evidence v2 contracts
- formal debate-agent handoff and prompt/tool-boundary rules
- a staged migration blueprint that keeps the current static path as baseline, fallback, and regression oracle

## Phases

### Phase 4: System Boundary Alignment

**Goal**: freeze the vocabulary, layer model, responsibilities, and non-goals for the expanded system  
**Depends on**: archived `v1.0` context  
**Plans**: 2 plans

Plans:

- [x] 04-01: Write the canonical expanded system definition, task entrypoint, and module-boundary baseline
- [x] 04-02: Lock the MCP-first tool stance, dynamic-SQL target path, and migration-era baseline rules

**Details:**

- Formalized the five-layer architecture as the single accepted system model
- Locked the formal task entrypoint to `person_id + policy_id + optional qualification scope`
- Chose MCP as the formal tool framing
- Positioned `text_to_sql` as the first runnable tool and static SQL as migration-period baseline only

### Phase 5: Database Cognition and Dictionary Memory Design

**Goal**: define how the system learns database meaning before generating evidence requests  
**Depends on**: Phase 4  
**Plans**: 2 plans

Plans:

- [x] 05-01: Formalize the ADC310-style dictionary-asset contract
- [x] 05-02: Formalize the task-level semantic packet produced by the cognition layer

**Details:**

- Turned the `ADC310.json` family into the model for a reusable dictionary contract
- Defined `source_refs`, `aliases`, and `notes` as first-class dictionary metadata
- Defined the semantic packet as a task-scoped structured bundle instead of a flat schema dump
- Locked excerpt-first dictionary loading and first-class `time_semantics`

### Phase 6: Evidence Planning and Dynamic Query Engine Contracts

**Goal**: define the canonical contracts for question templates, evidence plans, dynamic SQL generation, repair, and evidence objects  
**Depends on**: Phase 5  
**Plans**: 3 plans

Plans:

- [x] 06-01: Formalize qualification-item-first question templates and question-driven evidence plans
- [x] 06-02: Formalize the query chain from `query_intent` through generation, execution, bounded repair, and stop conditions
- [x] 06-03: Formalize Evidence v2 as a provenance-rich auditable artifact

**Details:**

- Defined reusable check-question templates tied to qualification items and conflict patterns
- Defined evidence-plan items as question-driven units with evidence targets, fields, time scope, priority, and missing-evidence handling
- Defined `query_intent` as the canonical intermediate artifact before Text-to-SQL
- Locked bounded repair and explicit stop conditions into the dynamic-query contract
- Elevated evidence from raw query output to Evidence v2 with planning, dictionary, execution, and repair provenance

### Phase 7: Agent Tool Boundary and Implementation Blueprint

**Goal**: define how agents consume the new planning/query layers and how future implementation will be staged  
**Depends on**: Phase 6  
**Plans**: 3 plans

Plans:

- [x] 07-01: Formalize the agent-facing evidence projection
- [x] 07-02: Formalize prompt loading rules and debate-time tool boundaries
- [x] 07-03: Formalize the staged implementation blueprint and deferred-scope list

**Details:**

- Made summary-first agent-facing evidence projection the default debate input
- Locked prompt loading into always-on, on-demand, and never-blindly-injected classes
- Formalized that debate agents do not directly call retrieval/query-repair tools during ordinary turns
- Preserved orchestrator-mediated future evidence-gap escalation
- Produced a staged migration plan that protects the current static chain as baseline, fallback, and regression oracle

## Milestone Summary

**Key Accomplishments:**

- Unified the expanded system under one five-layer architecture instead of scattered feature ideas
- Converted ADC310-style examples into a formal reusable dictionary contract
- Defined the task-scoped semantic packet and excerpt-first dictionary loading strategy
- Defined the canonical path from question template to evidence plan to `query_intent` to bounded dynamic query execution
- Defined Evidence v2 and the downstream summary-first evidence projection for debate agents
- Produced an implementation blueprint that lets later coding start without reopening the boundary debate

**Issues Resolved:**

- Ambiguity about what the expanded system actually is
- Ambiguity about how dictionaries, schema cognition, and task-time context differ
- Ambiguity about how future dynamic query generation connects to evidence planning and downstream debate
- Ambiguity about what debate agents should consume versus what tools/orchestrator should own

**Issues Deferred:**

- Implementing the first runnable `text_to_sql` slice
- Implementing `get_dict`, `plan_evidence`, and `auto_debug_sql`
- Building the post-rule validation layer
- Running large-scale evaluation beyond the current persona set
- Operationalizing learning-loop automation and concrete MCP deployment choices

**Technical Debt Accepted At Archive Time:**

- Milestone audit reported `gaps_found` because Phases 4-7 do not have formal `VERIFICATION.md` artifacts
- Phase 4 and Phase 5 summary files still lack `requirements_completed` frontmatter expected by the audit flow
- This milestone ends at design contracts, so runtime verification for the new architecture is intentionally deferred to the next implementation milestone
