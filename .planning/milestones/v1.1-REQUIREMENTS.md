# Requirements Archive: v1.1

**Archived:** 2026-03-24  
**Milestone:** v1.1  
**Type:** Design-only milestone  
**Core Value:** The system should produce grounded policy-eligibility decisions by preparing the right evidence before agent debate, keeping database semantics explicit enough to reduce hallucination and support auditability.

## Final Requirement Outcomes

### Alignment

- [x] **ALIGN-01**: Project has one agreed system definition for the expanded product, expressed as a five-layer architecture instead of scattered feature descriptions
- [x] **ALIGN-02**: Each newly proposed module has explicit inputs, outputs, responsibilities, and dependency boundaries
- [x] **ALIGN-03**: The milestone records explicit non-goals so future work does not silently expand from "designed" into "half-implemented architecture"

### Database Cognition

- [x] **SCHEMA-01**: Project defines the scope of the database-cognition layer, including tables, fields, key relations, aliases, code-value dictionaries, and temporal semantics
- [x] **SCHEMA-02**: Project defines the artifact contract for code dictionaries such as `dicts/ADC310.json`, including what metadata must exist and how future generation is expected to work
- [x] **SCHEMA-03**: Project defines how dictionary content is loaded on demand and partially injected into prompts instead of being blindly stuffed into every context window

### Evidence Planning

- [x] **PLAN-01**: Project defines the canonical structure of a reusable check-question template derived from policy clauses, qualification items, and conflict patterns
- [x] **PLAN-02**: Project defines the canonical structure of an evidence plan for `person_id + policy_id + qualification scope`, including evidence targets, relevant fields, time windows, priority, and missing-evidence rules
- [x] **PLAN-03**: Project defines how policy qualification items, question templates, and evidence-plan items connect to one another

### Dynamic Query Engine

- [x] **DSQL-01**: Project defines the contract for rewriting an evidence-plan item into a structured query intent that is suitable for runtime Text-to-SQL
- [x] **DSQL-02**: Project defines the contract for runtime Text-to-SQL generation, SQL execution, Auto Debugger behavior, and retry boundaries
- [x] **DSQL-03**: Project defines how generated SQL is logged, inspected, and traced without treating SQL strings as the system's long-term persistent memory
- [x] **DSQL-04**: Project defines fallback, failure, and stop conditions for cases where dynamic SQL is uncertain, broken, or semantically unsafe

### Evidence Object

- [x] **EVID-01**: Project defines an evidence-object v2 schema direction that supports planning, execution, debate, explanation, and later evaluation
- [x] **EVID-02**: Project defines how evidence artifacts record provenance such as dictionary references, query intent, execution status, time scope, and repair history

### Agent and Tool Boundary

- [x] **AGENT-01**: Project defines what the debate agents receive from the planning and query layers, and what they should no longer be expected to invent from raw database state
- [x] **AGENT-02**: Project defines first-class MCP-oriented tool contracts for dictionary lookup, evidence planning, query generation, and query repair, while allowing phased rollout instead of requiring all tools to ship at once
- [x] **AGENT-03**: Project defines prompt-context loading rules that distinguish always-present context, task-specific on-demand context, and context that should never be blindly injected

### Implementation Readiness

- [x] **READY-01**: Project ends with a phased implementation roadmap that can be executed later without reopening the system boundary debate from scratch
- [x] **READY-02**: Project explicitly records what is deferred beyond this milestone, especially rule-engine implementation, evaluation execution, and continuous-learning automation

### Deferred Beyond v1.1

- [ ] **RULE-01**: System implements a post-debate rule-engine validation layer against the new evidence contracts
- [ ] **EVAL-01**: System defines and runs large-scale evaluation cases beyond the current small persona set
- [ ] **LEARN-01**: System operationalizes retained repair artifacts, failure labels, and reusable few-shot memory into a real learning loop
- [ ] **MCP-01**: System exposes the finalized tool contract through a concrete MCP server implementation once the abstraction is stable
- [ ] **MIG-01**: System safely migrates from the current static SQL-template baseline to the new dynamic query path with regression safeguards
