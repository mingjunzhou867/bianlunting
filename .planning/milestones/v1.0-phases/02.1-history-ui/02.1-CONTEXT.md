# Phase 2.1: History UI Integration - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase wires the existing frontend to the saved-session APIs so users can browse prior debate sessions inside the current app shell. It does not redesign the whole product, add auth/governance, or replace the live debate submission flow. The boundary is limited to showing saved history on the left, rendering one selected saved session in the center, and keeping live submission usable for the current `id_card`.

</domain>

<decisions>
## Implementation Decisions

### App shell layout
- The left pane should become the saved-session history list rather than the evidence board
- The center pane should remain the primary renderer for evidence, round history, and final conclusion
- Saved-session evidence should move into the center-pane detail view so historical and live sessions share one visual surface
- The first version should stay inside the existing single-page app rather than introducing a separate route or page

### History loading scope
- History list data should be loaded for the currently submitted `id_card`
- The first version should not add a global "recent sessions" browser
- Submitting a new `id_card` should refresh the list for that person
- The list should use the existing summary-only backend response from `GET /api/debates?id_card=...`

### Live versus historical state model
- Clicking a history item should fetch `GET /api/debates/{session_id}` and hydrate the center pane from the saved snapshot
- Submitting an `id_card` should still run the live `/api/debate_stream` flow and show live agent progress in the center pane
- After a live debate completes, the history list for that `id_card` should refresh so the new saved session becomes visible immediately
- The center renderer should be shared by both live data and saved-session detail instead of maintaining two separate detail UIs

### Empty, loading, and selection behavior
- If no `id_card` has been submitted and no history item is selected, the center pane should stay empty
- If an `id_card` has history but nothing is selected yet, the newest history item should be auto-selected in the first version
- History loading failure should be non-blocking for the rest of the page and surfaced as a clear frontend message
- Session-detail loading should show a local loading state without wiping the whole app shell

### Claude's Discretion
- Exact wording for history list labels, badges, and empty-state copy
- Whether the selected-session visual treatment is a card highlight, active row border, or similar affordance
- Minor formatting choices for timestamps and consensus metadata as long as the list stays compact

</decisions>

<specifics>
## Specific Ideas

- The left-side history rows should stay compact and summary-only: conclusion, stance, completion time, source endpoint, and basic consensus/round metadata
- The center pane should be able to render both a just-finished live debate and a previously saved session with no special-casing visible to the user
- Newly completed live sessions should appear at the top of the history list after refresh because backend history is already newest-first
- This phase should favor low-risk reuse of the current `App.vue` renderer over a broader frontend redesign

</specifics>
