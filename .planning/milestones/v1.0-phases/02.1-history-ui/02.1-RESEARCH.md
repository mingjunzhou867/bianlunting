# Phase 2.1: History UI Integration - Research

**Researched:** 2026-03-23
**Status:** Ready for planning

## Goal

Research how to surface saved debate sessions in the existing Vue app without breaking the live streaming workflow already used by the demo.

## Findings

### 1. Current frontend architecture is single-page and App.vue-centric

- [frontend/src/App.vue](/c:/Users/afrangry/PycharmProjects/bysj_t2s/frontend/src/App.vue) currently owns almost all page state, layout, and live-stream event handling
- There is no router, store, or shared API client layer yet, so the lowest-risk implementation path is to keep this phase inside the existing single-page shell
- The left pane is currently dedicated to `EvidenceBoard`, while the center pane already renders round history and final outcome

### 2. Saved-session detail already matches the center-pane shape closely

- The live UI builds around a shape like `{ id_card, evidence, debate: { history, final_conclusion, final_stance, rounds_taken, consensus_rate, is_consensus_reached } }`
- Phase 2's snapshot-first detail API returns the same core evidence-plus-history structure, so a shared center renderer is practical without rebuilding the presentation model
- [frontend/src/components/AgentCard.vue](/c:/Users/afrangry/PycharmProjects/bysj_t2s/frontend/src/components/AgentCard.vue) and [frontend/src/components/EvidenceBoard.vue](/c:/Users/afrangry/PycharmProjects/bysj_t2s/frontend/src/components/EvidenceBoard.vue) are reusable if the app normalizes live and saved data into one state shape

### 3. The main implementation risk is state collision, not API shape

- The backend already exposes:
  - `GET /api/debates?id_card=...` for compact history rows
  - `GET /api/debates/{session_id}` for snapshot-first detail
  - `POST /api/debate_stream` for live SSE updates
- The bigger frontend risk is preventing live-stream updates, saved-session detail loads, and selection changes from stomping on each other inside one `currentData` object
- The safest structure is to separate:
  - current query identity (`id_card`)
  - history list state
  - selected session identity
  - center-pane session payload
  - loading/error state for history, detail, and live run independently

### 4. Verification should stay lightweight in this phase

- [frontend/package.json](/c:/Users/afrangry/PycharmProjects/bysj_t2s/frontend/package.json) currently has no test runner, no Vue Test Utils, and no test scripts
- Adding a full frontend test stack is possible, but it would expand scope and overlap with Phase 3 hardening
- The most phase-appropriate verification target is:
  - successful `vite build`
  - focused manual checks for empty state, history selection, live run, and post-run history refresh

## Recommendations

### Recommended implementation shape

- Keep the app single-page for this phase
- Extract a shared center-pane renderer component instead of duplicating a "history detail" view beside the live renderer
- Move `EvidenceBoard` into the center-pane session view so the left pane can become the history list
- Keep API calls near `App.vue` unless the file becomes unmanageable enough to justify a tiny helper module

### Recommended plan split

- Plan `02.1-01`: refactor page state and center-pane rendering so one view can show either live or saved session data
- Plan `02.1-02`: wire history list fetching, session selection, and post-stream refresh on top of that shared state model

## Risks To Watch

- Saved-session detail loads racing with a newly started live stream
- Sidebar history fetch failure wiping the selected center-pane content
- Overgrowing `App.vue` if extraction boundaries are not chosen early
- Regressing the current SSE experience while adding non-stream history reads

## Planning Implications

- Plans should preserve the existing `/api/debate_stream` submission path rather than replacing it with a non-stream flow
- Plans should avoid introducing a router, global store, or broad redesign in this phase
- Plans should explicitly account for separate loading/error states for:
  - history list
  - saved-session detail
  - live streaming run
