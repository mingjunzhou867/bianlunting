# Milestone v1.0: Debate Persistence and Retrieval

**Status:** SHIPPED 2026-03-24  
**Phases:** 1-3  
**Total Plans:** 9

## Overview

This milestone transformed the project from a live-only debate experience into a saved-session product slice. Completed debates now persist to MySQL, can be retrieved by `id_card` or `session_id`, render inside the frontend history UI without rerunning agents, and have focused regression/documentation coverage around the new save/read/render chain.

## Phases

### Phase 1: Persistence Foundation

**Goal**: Define and implement a stable persistence model for completed debate sessions  
**Depends on**: Nothing  
**Plans**: 3 plans

Plans:

- [x] 01-01: Audit current debate persistence path and define canonical storage contract
- [x] 01-02: Implement MySQL schema changes and backend write-layer updates
- [x] 01-03: Integrate complete-session persistence into both debate endpoints and surface failures clearly

**Details:**

- Successful completed sessions are the only persistence target
- Runtime persistence and MySQL DDL were aligned
- A canonical `debate_session` table plus structured `agent_debate_log` rows were introduced
- Persistence failures were made explicit instead of silent

### Phase 2: Retrieval APIs

**Goal**: Provide stable backend read APIs for saved debate sessions  
**Depends on**: Phase 1  
**Plans**: 2 plans

Plans:

- [x] 02-01: Build read/query service that reconstructs one saved debate session and session lists
- [x] 02-02: Expose retrieval endpoints and align DTOs/contracts with existing consumers

**Details:**

- Added history list retrieval by `id_card`
- Added snapshot-first detail retrieval by `session_id`
- Preserved `/api/debate` and `/api/debate_stream` compatibility

### Phase 2.1: History UI Integration (INSERTED)

**Goal**: Let the frontend browse saved debate sessions while keeping live debate submission intact  
**Depends on**: Phase 2  
**Plans**: 2 plans

Plans:

**Details:**

## Milestone Summary

**Decimal Phases:**

- Phase 2.1: History UI Integration (inserted before hardening so retrieval and UI landed in one milestone slice)

**Key Accomplishments:**

- Established one canonical persistence contract shared by runtime code and MySQL DDL
- Persisted completed sessions from both `/api/debate` and `/api/debate_stream`
- Exposed stable retrieval APIs for history lists and snapshot-first detail
- Surfaced saved-session browsing inside the existing frontend shell
- Added backend regression checks and a lightweight frontend smoke test for the new chain
- Created maintainer-facing documentation for saved-session setup and operation

**Issues Resolved:**

- Persistence/schema drift between runtime code and `agent_debate_log`
- Lack of a queryable saved-session model for completed debates
- No frontend path for browsing saved debate history without rerunning agents

**Issues Deferred:**

- Authorization and audit policy for viewing saved sessions
- Retention and cleanup rules for saved-session data
- Richer history browsing such as filters, global recent sessions, or comparison views

**Technical Debt Accepted At Archive Time:**

- Milestone audit reported `gaps_found` because phase-level `VERIFICATION.md` files and summary `requirements-completed` frontmatter were not created
- Frontend production build still reports a non-blocking large-chunk warning
