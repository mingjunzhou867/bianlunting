DIN-SQL：作为 Text-to-SQL 主框架
论文：DIN-SQL: Decomposed In-Context Learning of Text-to-SQL with Self-Correction。
它把 Text-to-SQL 拆成 4 个可控步骤，非常适合你现在这种业务表有限、证据链固定的场景。原论文和代码都能找到。

ReAct：作为 Agent 调工具的交互范式
论文：ReAct: Synergizing Reasoning and Acting in Language Models。
你可以让主 Agent 不直接“空想结论”，而是先输出要验证的资格项，再调用 SQL 工具取证，再根据 observation 更新判断。这个模式和你的“数据库事实驱动”高度一致。

Reflexion：作为错误修正/复盘机制
论文：Reflexion: Language Agents with Verbal Reinforcement Learning。
它不是改模型权重，而是用语言反思来修正后续行为。你可以把它改造成“SQL 失败反思”或“结论-证据不一致反思”。

这三篇组合起来，不但能做出系统，而且很好写进论文或比赛文档：
DIN-SQL 负责生成证据查询，ReAct 负责流程编排，Reflexion 负责错误修正。

为什么我不建议你先复现“完整多智能体辩论”

因为从落地性看，多智能体辩论最贵、最难控、最不稳定。
2023 年那篇经典 MAD 论文确实证明了，多模型实例之间的 propose-and-debate 可以提升推理和 factuality。它很适合作为你的理论支撑。

但到了 2025 年，Debate or Vote 这篇工作进一步分析发现：MAD 的很多收益，其实主要来自 majority voting，而不是 debate 本身；简单集成方法在很多场景里更稳定。对你来说，这个结论很重要，因为它意味着你可以先做：
多个视角独立初判 + 审计 Agent 反驳 + 最终投票/裁决，而不是先做 2–3 轮长辩论。

所以如果你要写“多 Agent”，我建议第一版不要做成：

每轮都自由攻击别人

最后再投票

而应该做成：

1 个审计 Agent 专门找漏洞

必要时触发 1 次修正

最后输出“符合 / 不符合 / 数据缺失”

这已经足够体现“多智能体协作”了，而且工程复杂度低很多。

第 1 步：资格项拆解
把“是否符合某项补贴资格”拆成若干证据问题。
例如：

第 2 步：证据项到表字段映射
这里相当于 DIN-SQL 的 schema linking。
不用做通用 linking，直接做：

资格项 → 相关表

资格项 → 关键字段

资格项 → 时间字段

资格项 → 缺失判定规则

第 3 步：生成模板 SQL
不要先追求自由生成 SQL。
你先做模板化 SQL 拼接，再让 LLM 只补 WHERE 条件、JOIN 路径、时间窗和聚合逻辑。

第 4 步：self-correction
第一次 SQL 执行失败，就把报错和 schema 回喂给模型，让它重写一次。
这一步就是 DIN-SQL / Reflexion 思想最容易落地的地方。

为什么这条路最适合你

因为它避开了大部分 Text-to-SQL 最难的问题：
你不是开放域问答，不需要对任意自然语言都泛化；你是固定业务语义 → 固定证据查询，这比 Spider 式通用 Text-to-SQL 简单很多。Text-to-SQL 综述也一直把 schema linking、复杂 SQL、跨域泛化列为关键难点，而你的场景恰好可以通过领域收窄来降低难度。

你可以设计 3 个角色，而不是 5 个：

否则多数决

Text-to-SQL 这块，你该选哪种技术深度

这里我给你一个非常明确的判断：

第一阶段：不要上 PICARD / RAT-SQL 训练复现

PICARD 和 RAT-SQL 都是非常经典的 Text-to-SQL 工作。
RAT-SQL 强在 relation-aware schema encoding 与 schema linking；PICARD 强在 constrained decoding，能减少无效 SQL 生成。它们都很有学术价值。

但是对你现在来说，这两条都不适合作为第一落地点：

RAT-SQL 更像完整模型训练路线，训练和适配成本高。

PICARD 更适合你已经决定走 T5 类生成模型并做正式 Text-to-SQL 训练时再加。

第一阶段最适合你的方式

模板 SQL + LLM 填槽 + 执行反馈纠错。

这个思路虽然没那么“炫”，但最可能在你的项目里成功，而且理论上能挂靠：

DIN-SQL 的 task decomposition

ReAct 的工具调用式推理

Execution-guided 的执行反馈思想

Reflexion 的语言反思纠错

也就是说，你完全可以把自己的系统写成：

提示词方面，你现在最该吸收哪几类论文

不是所有 prompting 都值得你看。对你最有价值的是这几类：

1. CoT

经典基础。让模型显式输出中间推理步骤。

2. Least-to-Most

特别适合“复杂资格认定拆成若干子资格项”的场景。
它的思想就是先拆子问题，再逐步求解。和你项目几乎天然匹配。

3. Self-Consistency

如果你确实想保留一点“多样本思考”，比起复杂多 Agent，先做 self-consistency 更省成本。即同一个问题采样多条 reasoning path，再选一致答案。

4. Reflexion

它比纯 CoT 更适合系统工程，因为你是有外部反馈的：SQL 执行结果、规则校验结果、审计驳回结果。
